# requirements.md

## 概要

YouTube Data API v3 のクォータ超過エラー（403 quotaExceeded）による処理クラッシュと、
無意味なリトライループを解消する。
API 呼び出し回数を削減するアルゴリズムに変更する。

---

## 背景・現状の問題

### 問題 1: クォータ超過時にクラッシュする

以下の API 呼び出しが `api_call_with_retry` を使っていないため、
クォータ超過（403）が発生すると例外がそのまま伝播してアプリがクラッシュする。

| 関数 | 呼び出し場所 | 未対応の理由 |
|---|---|---|
| `fetch_playlists` | 起動直後（プレイリスト選択） | 当初から未対応 |
| `fetch_playlist_video_ids` | `process_add` 実行時 | 当初から未対応 |

> **実装中に発覚した追加バグ（2026-06-26）:**
> 当初 `fetch_playlist_video_ids` のみを修正対象としていたが、
> 実行時に `fetch_playlists`（起動直後のプレイリスト一覧取得）でも
> 同じクラッシュが発生することが判明した。
> また `controller/cli.py` の `choose_playlists` 呼び出しが `try` ブロックの外にあり、
> `QuotaExceededError` が `main()` でキャッチされない問題も同時に修正した。

### 問題 2: リトライが機能しない

`api_call_with_retry` は 60 秒待ってリトライするが、
YouTube API のクォータは **太平洋時間の午前 0 時（日次）にリセット**される。
60 秒では解消されないため、無限ループに陥り実質的に使えない。

### 問題 3: 毎回プレイリスト全件を API で取得している

`process_add` のたびに `fetch_playlist_video_ids` を呼び出してプレイリスト全件を取得する。
プレイリストが大きいほど消費クォータが増加する。

## YouTube Data API v3 クォータ仕様（参考情報）

> 出典: [YouTube Data API Overview](https://developers.google.com/youtube/v3/getting-started) /
> [Quota Calculator](https://developers.google.com/youtube/v3/determine_quota_cost) /
> [PlaylistItems: insert](https://developers.google.com/youtube/v3/docs/playlistItems/insert) /
> [PlaylistItems: delete](https://developers.google.com/youtube/v3/docs/playlistItems/delete) /
> [PlaylistItems: list](https://developers.google.com/youtube/v3/docs/playlistItems/list)

### 1日のクォータ上限

| 項目 | 値 |
|---|---|
| デフォルト上限 | **10,000 ユニット / 日 / プロジェクト** |
| リセット時刻 | **太平洋時間（PT）の午前 0 時** |
| 複数 API キーの扱い | 同一プロジェクト内で共有（合計 10,000 ユニット） |
| 上限引き上げ | Google へのコンプライアンス審査申請が必要 |

※ 2025年12月4日の改定で `videos.insert` のコストが約 1,600 → 約 100 ユニットに変更された。

### 本アプリで使用するメソッドのコスト

| メソッド | コスト | 1日の最大呼び出し回数 |
|---|---|---|
| `playlistItems.list` | **1 ユニット** / ページ（50件） | 10,000 回 |
| `playlistItems.insert` | **50 ユニット** | **200 回** |
| `playlistItems.delete` | **50 ユニット** | **200 回** |
| `playlists.list` | **1 ユニット** | 10,000 回 |

### クォータ消費シミュレーション

#### 具体例: プレイリスト 100 件 → 40 件追加

| フェーズ | メソッド | 計算 | ユニット |
|---|---|---|---|
| 既存取得 | `playlistItems.list` | ceil(100 / 50) = 2 ページ | 2 |
| 動画追加 | `playlistItems.insert` | 40 件 × 50 | 2,000 |
| **合計（現状）** | | | **2,002** |

**1日に実行できる回数（現状）:**

```
10,000 ÷ 2,002 ≈ 4 回
```

→ 同じ操作（40 件追加）を 1 日に 4 回まで実行可能（4 回目終了時 8,008 ユニット消費）

---

**キャッシュ導入後（2 回目以降はキャッシュヒット）:**

| フェーズ | メソッド | 計算 | ユニット |
|---|---|---|---|
| 既存取得 | キャッシュヒット | スキップ | **0** |
| 動画追加 | `playlistItems.insert` | 40 件 × 50 | 2,000 |
| **合計（改善後）** | | | **2,000** |

```
10,000 ÷ 2,000 = 5 回
```

→ 同じ操作を **1 日に 5 回**実行可能（追加できる総動画数: 200 件）

#### 途中でクォータ超過した場合の挙動

例: 40 件追加しようとして 20 件目で quotaExceeded が発生した場合

```
現状:  クラッシュ → 再実行時に既存 120 件分を再取得（2 ユニット消耗）
改善後: グレースフルに終了 → キャッシュに追加済み 20 件を反映
        → 翌日再実行時、キャッシュから既存 120 件を読み込み（0 ユニット）
        → 残り 20 件の追加のみ実行（1,000 ユニットで完了）
```

---

### 1日に追加できる動画本数

#### 計算

```
1日のクォータ上限:          10,000 ユニット
playlists.list（起動時）:  -      1 ユニット（1回）
残り:                        9,999 ユニット

1動画の追加コスト:              50 ユニット

9,999 ÷ 50 = 199.98 → 最大 199 本/日
```

#### 一括 vs 複数回に分けた場合

| シナリオ | 消費ユニット | 追加可能本数 |
|---|---|---|
| 1回で一気に追加（キャッシュあり） | 1 + M×50 | **最大 199 本** |
| 40本 × 複数回（キャッシュあり） | N + 40×50×N | **4回まで（160本）**、5回目で超過 |
| 40本 × 複数回（キャッシュなし） | 3×N + 40×50×N | 同じく4回まで（誤差レベル） |

#### 一括追加（バッチ）について

YouTube Data API v3 には複数動画を一度に追加するエンドポイントが存在しない。
`playlistItems.insert` は **1動画 = 1リクエスト（50ユニット固定）** が仕様。

Google API の Batch HTTP Request（複数リクエストを1つのHTTP通信にまとめる機能）は存在するが、
**クォータ消費は変わらない**（N×50ユニット）。ネットワーク往復が減るだけ。

```
通常:  N回のHTTPリクエスト → N × 50 ユニット
Batch: 1回のHTTPリクエスト → N × 50 ユニット（変わらない）
```

→ バリデーション後にまとめて追加する設計にしても、クォータ節約にはならない。
  今回のキャッシュ設計が実質的にできる最大の削減策。

## ユーザーストーリー

- ユーザーとして、クォータが超過しても **アプリがクラッシュせず** 明確なメッセージで終了してほしい
- ユーザーとして、**無駄な API 呼び出しを減らす**ことで、クォータ上限に達するまでの動画追加件数を増やしてほしい
- ユーザーとして、クォータが超過した際に**いつ再実行すればよいか**がわかるメッセージを見たい

---

## 受け入れ条件

1. `fetch_playlists`・`fetch_playlist_video_ids` のいずれでクォータ超過が発生しても、クラッシュせず適切なメッセージを出して終了する
2. クォータ超過時に「日次リセット（太平洋時間 0:00）まで X 時間」を表示する
3. プレイリストの既存動画 ID を **ローカルにキャッシュ**し、キャッシュが有効な間は API 呼び出しをスキップする
4. キャッシュは追加・削除操作の成功後にインクリメンタル更新する（再取得不要）
5. キャッシュが存在しない、または TTL を超過した場合のみ API で取得する
6. キャッシュの TTL はデフォルト 23 時間（環境変数 `CACHE_TTL_HOURS` で上書き可能）

---

## 制約事項

- キャッシュファイルは既存の `footprints/youtube/` ディレクトリ配下に置く
- キャッシュはプレイリスト ID ごとに管理する（複数プレイリスト対応）
- `api_call_with_retry` の無限リトライは廃止し、クォータ超過時は即時終了とする
- テストでは実 API を呼ばず、既存のモック方針を踏襲する
